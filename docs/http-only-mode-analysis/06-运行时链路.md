# 运行时链路

## 1. runtime-http 的定位

`packages/runtime-http` 是 `only_http` 模式的运行时承接层。

它不负责：

- 标准 protobuf 二进制编解码
- 标准 unary / streaming RPC 生命周期

它负责的是：

- 保存服务与方法元信息
- 在调用时把方法元信息翻译成 HTTP 请求
- 委托给外部注入的 HTTP 实现

## 2. ServiceType

`runtime-http/src/service-type.ts` 中的 `ServiceType` 很薄。

它的核心职责只有两个：

1. 保存 `typeName`
2. 把 `PartialMethodInfo[]` 规范化成 `MethodInfo[]`

也就是说，它更像“服务描述对象”，不是“调用引擎”。

## 3. MethodInfo

`runtime-http/src/reflection-info.ts` 中定义了 `MethodInfo`。

它至少包含：

- `service`
- `name`
- `localName`
- `idempotency`
- `I`
- `options`

在 `only_http` 模式里，运行时最依赖的是：

- `method.options["google.api.http"]`

因为 URL、HTTP Method、上传判定都从这里取。

## 4. HttpTransport 的职责

真正执行请求的是：

- `runtime-http/src/rpc-transport.ts`

`HttpTransport` 构造函数接受两种输入：

- 一个 `VAxios` 构造器
- 一个现成的 `VAxiosInstance`

这样调用方既可以传类，也可以传实例。

## 5. request() 的整体流程

生成出来的 client 方法，最终都会走到：

```ts
HttpTransport.request(method, input, opt)
```

其内部流程可以概括为：

1. 调 `makeUrl(method)` 计算 URL
2. 调 `makeMethod(method)` 计算 HTTP Method
3. 调 `isUpload(method)` 判断是否上传
4. 调 `vAxios.request()` 或 `vAxios.uploadFile()`

## 6. URL 的来源

`makeUrl()` 的逻辑很直接：

- 如果存在 `google.api.http`
  - 取 `get / post / put / delete / patch` 中实际存在的那个路径
- 否则退回到默认的 `service.typeName/methodName`

因此在正常 `only_http` 用法下，URL 基本完全由 proto 注解决定。

## 7. HTTP Method 的来源

`makeMethod()` 的逻辑也是直接映射：

- `get -> GET`
- `post -> POST`
- `put -> PUT`
- `delete -> DELETE`
- `patch -> PATCH`

这一步没有做更复杂的协议推导。

## 8. 上传判定

是否走上传逻辑，由下面规则决定：

- `method.options["google.api.http"].body === "file"`

如果是上传：

- 调用 `vAxios.uploadFile()`
- 自动加 `multipart/form-data;charset=UTF-8`

如果不是上传：

- 调用 `vAxios.request()`

## 9. 对外部 HTTP 实现的依赖

`runtime-http` 自己不实现真正的网络库。

它依赖调用方传入一个满足如下接口的对象：

```ts
interface VAxiosInstance {
  request<T>(config, options?): HttpResult<T>;
  uploadFile<T>(config, params?): HttpResult<T>;
}
```

也就是说，它本质上只是一个 transport adapter。

## 10. 返回值模型

`runtime-http/src/http-result.ts` 里把返回值定义成了业务包装结构：

- `code`
- `msg`
- `data`
- `errors`

这说明这个 transport 不是“通用 protobuf RPC 结果模型”，而是“面向具体业务风格的 HTTP 返回包装”。

## 11. 运行时链路总结

`only_http` 的运行时调用路径可以简化成：

1. 业务调用生成的 client 方法
2. 方法内部找到对应 `MethodInfo`
3. `HttpTransport` 从 `google.api.http` 中提取 URL 和 HTTP 动词
4. 决定是普通请求还是上传
5. 委托给外部 `VAxios`

所以运行时真正的关键不是 protobuf 编码，而是“基于方法元信息进行 HTTP 路由分发”。
