# 设计取舍与问题

## 1. 这个模式的优势

从工程实践角度看，`only_http` 模式有几项明确优势。

### 1.1 契约统一

前后端都可以围绕同一份 proto 协议协作。

proto 不再只描述消息结构，也同时描述：

- 服务方法
- URL
- HTTP Method

### 1.2 前端接入轻

生成产物偏轻量：

- 有类型
- 有客户端
- 没有沉重的标准 RPC 运行时依赖链

这对前端 SDK 场景很友好。

### 1.3 易于平台化

如果团队内部已经约定：

- 用 proto 定义接口
- 用 `google.api.http` 描述 REST 路由
- 用统一 HTTP 库发请求

那么这个模式很适合做统一 SDK 基建。

## 2. 这个模式的边界

虽然它实用，但能力边界也很明显。

### 2.1 不是标准 protobuf over HTTP

当前实现并没有按标准 protobuf / grpc 风格去编码消息，而是直接围绕普通 HTTP 请求封装。

### 2.2 对 google.api.http 的支持不完整

从源码上看，目前主要使用了：

- `get`
- `post`
- `put`
- `delete`
- `patch`
- `body`

没有看到对以下能力的完整支持：

- 路径模板变量替换
- `additional_bindings`
- 更细粒度的 body 字段映射

### 2.3 上传能力是项目私有约定

`body === "file"` 被当成上传判定规则。

这很实用，但它更像团队约定，而不是标准 `google.api.http` 的通用语义。

### 2.4 偏业务响应模型

`HttpResult<T>` 并不是返回裸 `T`，而是返回：

- `code`
- `msg`
- `data`
- `errors`

这说明 `runtime-http` 已经带有明显业务风格。

## 3. 源码中值得注意的问题

下面这些不是风格问题，而是实打实的实现不一致点。

### 3.1 ServiceTypeGeneratorHttp 里 O 类型写错

在 `packages/plugin/src/code-gen/http-type-generator.ts` 中，输出 `O.typeName` 时实际取的是 `methodInfo.I.typeName`。

结果就是生成的 `service.ts` 中，输出类型元信息会写错。

这个问题在 `packages/example-http/service.ts` 的生成结果里已经能看到。

### 3.2 runtime-http 的 MethodInfo 类型没有 O

`runtime-http/src/reflection-info.ts` 中的 `MethodInfo` 只显式定义了 `I`，没有定义 `O`。

但生成器和生成结果却依然在写 `O` 字段。

这说明编译期元信息模型和运行时类型定义存在漂移。

### 3.3 普通请求统一走 params

`HttpTransport.request()` 里普通请求直接使用：

```ts
params: input
```

这意味着它没有根据 `body: "*"` 去区分 query/body 语义，而是把解释责任更多留给下层 HTTP 库或团队约定。

### 3.4 聚合 http-client.ts 的 header 信息异常

从生成结果看，`http-client.ts` 的文件头信息并不是来自正常的 `FileDescriptorProto`，而更像误用了 `ServiceDescriptorProto`。

这说明聚合文件的生成上下文存在不一致。

## 4. 怎么理解这些问题

这些问题说明：

- `only_http` 很像一个“面向业务落地的定制分支”
- 它是可用的，但并不像仓库中标准 RPC 能力那样完整和统一

也就是说，这个模式目前更适合作为：

- 内部 HTTP SDK 生成方案

而不是：

- 严格的通用 protobuf HTTP 标准实现

## 5. 最终结论

如果从架构目标上评价，`only_http` 模式是清晰的：

- 它要解决的是“用 proto 生成前端 HTTP SDK”

如果从抽象完整度上评价，它仍然是偏定制化、偏业务化的实现：

- 优点是轻、直接、好接入
- 代价是标准语义覆盖有限，编译期与运行时模型还有一些不一致点
