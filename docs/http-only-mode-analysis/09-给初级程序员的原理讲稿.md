# 给初级程序员的 only_http 原理讲稿

这篇文档的目标不是让你背 API，而是让你能把 `only_http` 的原理讲清楚。

读完以后，你应该能回答这些问题：

1. `protobuf-ts` 为什么要分“编译期”和“运行时”？
2. `only_http` 为什么不是一个普通 axios 封装？
3. `google.api.http` 在这条链路里到底起什么作用？
4. 生成出来的 `service.ts`、`service.client.ts`、`http-client.ts` 分别负责什么？
5. 一次业务调用从 `httpClient.rum.report(input)` 到真正发 HTTP 请求，中间经过了哪些对象？
6. 它和标准 gRPC / grpc-web / Twirp 模式有什么本质差别？

---

## 1. 先用一句话建立心智模型

`only_http` 可以理解成：

> 把 `.proto` 里的 service 方法和 `google.api.http` 路由注解，翻译成 TypeScript 里能直接调用的 HTTP SDK。

这句话里有三个关键词。

第一个关键词是 `.proto`。它是接口契约，描述有哪些消息、枚举、服务和方法。

第二个关键词是 `google.api.http`。它把一个 RPC 方法映射到普通 HTTP 路由，比如 `POST /v1/rum/report`。

第三个关键词是 HTTP SDK。最终业务代码不用自己拼 URL，也不用手写每个接口函数，而是直接调用生成好的 TypeScript client。

---

## 2. 为什么不是直接手写 HTTP Client

如果只有一两个接口，手写当然可以：

```ts
axios.post("/v1/rum/report", input);
```

但接口一多，就会出现几个问题：

- URL 写在前端代码里，后端改路由时前端很难同步。
- 请求参数和响应类型靠人工维护，容易漂移。
- 多个项目都要写同样的 client 封装。
- 接口定义、类型定义、调用代码散落在不同地方。

`only_http` 想解决的是这些工程问题：

- 用 `.proto` 统一保存接口契约。
- 用 `google.api.http` 统一保存 HTTP 路由。
- 用代码生成器稳定生成 TypeScript 类型和 client。
- 用 `runtime-http` 在运行时根据元信息发请求。

所以它不是“为了少写几行 axios”，而是为了让接口契约、类型和调用入口由同一个源头生成。

---

## 3. 先分清编译期和运行时

理解 `protobuf-ts` 最容易卡住的地方，是把编译期和运行时混在一起。

### 3.1 编译期做什么

编译期发生在执行 `protoc` 的时候。

大致流程是：

```text
.proto 文件
  -> protoc 解析
  -> CodeGeneratorRequest
  -> @protobuf-ts/plugin
  -> 生成 .ts 文件
```

这里的核心问题是：

> 怎么把 proto 描述符变成 TypeScript 代码？

在 `only_http` 模式里，编译期会生成这些东西：

- 消息接口和枚举，比如 `ReportRequest`、`ReportReply`。
- HTTP 专用服务元信息，比如 `Rum` 这个 `ServiceType`。
- HTTP client，比如 `RumClient`。
- 聚合入口，比如 `HttpClient`。

### 3.2 运行时做什么

运行时发生在业务代码真正调用接口的时候。

大致流程是：

```text
业务代码调用 httpClient.rum.report(input)
  -> 生成的 RumClient.report()
  -> HttpTransport.request(method, input, options)
  -> 根据 google.api.http 取 URL 和 Method
  -> 调用外部注入的 vAxios.request() 或 vAxios.uploadFile()
```

这里的核心问题是：

> 怎么把一次 TypeScript 方法调用变成一次 HTTP 请求？

`runtime-http` 不负责生成代码，也不负责解析 `.proto` 文件。它只接收生成物里携带的服务方法元信息，然后把这些元信息翻译成 HTTP 请求配置。

---

## 4. only_http 是怎么被打开的

示例工程里通过 `--ts_opt=only_http` 打开这个模式。

你可以看：

- `packages/example-http/Makefile`
- `packages/plugin/src/our-options.ts`
- `packages/plugin/src/protobufts-plugin.ts`

关键点是：

```text
--ts_opt=only_http
  -> options.onlyHttp = true
  -> 插件进入 HTTP 专用生成分支
```

在 `packages/plugin/src/protobufts-plugin.ts` 里，普通模式和 `only_http` 模式会走不同生成器。

普通模式大致会生成：

- `MessageType`
- 标准 `ServiceType`
- generic client
- grpc client
- server 定义

`only_http` 模式大致会生成：

- 消息接口和枚举
- HTTP 专用 `ServiceType`
- HTTP client
- 额外的 `http-client.ts` 聚合入口

这说明 `only_http` 不是一个独立插件，而是同一个插件里的一个生成分支。

---

## 5. google.api.http 为什么是核心

先看一个 proto 方法：

```proto
rpc Report(ReportRequest) returns(ReportReply) {
  option (google.api.http) = {
    post: "/v1/rum/report"
    body: "*"
  };
}
```

如果没有 `google.api.http`，这个方法只表达了：

```text
ReportRequest -> ReportReply
```

也就是说，它告诉我们“输入是什么、输出是什么”，但没有告诉普通 HTTP 世界里的：

- URL 是什么？
- HTTP Method 是 GET 还是 POST？
- 请求体怎么处理？

加上 `google.api.http` 后，它就多了 HTTP 路由语义：

```text
ReportRequest -> POST /v1/rum/report -> ReportReply
```

`only_http` 正是依赖这部分信息，才能把一个 proto service 方法变成前端能直接调用的 HTTP 方法。

---

## 6. 编译期第一步：protoc 不把 proto 文本直接交给插件

初学者容易误以为插件是在读 `.proto` 文本。实际不是。

`protoc` 会先解析 `.proto`，然后把结构化的描述符发给插件。插件拿到的是 `CodeGeneratorRequest`，不是原始文本。

这很重要，因为描述符里已经把 proto 信息拆成了结构化对象：

- 文件描述符
- 消息描述符
- 枚举描述符
- 服务描述符
- 方法描述符
- options

所以 `protobuf-ts` 的生成器不是靠字符串搜索 `rpc Report`，而是遍历描述符对象。

可以这样讲：

> protoc 像编译器前端，负责把 proto 文本解析成 AST/描述符；`@protobuf-ts/plugin` 像编译器后端，负责把描述符翻译成 TypeScript 代码。

---

## 7. 编译期第二步：Interpreter 把描述符解释成更好用的模型

描述符很完整，但直接用起来不方便。

比如一个 service method 里会有：

- 方法名
- 输入类型名
- 输出类型名
- options
- stream 信息
- 注释信息

生成器真正想要的是更顺手的结构，例如：

```text
MethodInfo
  name
  localName
  I.typeName
  O.typeName
  options["google.api.http"]
```

`Interpreter` 的作用就是把 protobuf descriptor 解释成生成器容易消费的服务模型。

换句话说：

> Descriptor 是 protoc 给的原始结构化信息，Interpreter 是 protobuf-ts 对这些信息做的二次整理。

---

## 8. 编译期第三步：生成 HTTP ServiceType

`only_http` 模式会用：

- `packages/plugin/src/code-gen/http-type-generator.ts`

它生成的结果类似：

```ts
export const Rum = new ServiceType("rum.Rum", [
  {
    name: "Report",
    options: { "google.api.http": { post: "/v1/rum/report", body: "*" } },
    I: { typeName: "ReportRequest" },
    O: { typeName: "ReportReply" }
  }
]);
```

这个文件非常关键，因为它把 proto 方法上的 HTTP 注解保存到了 TypeScript 运行时对象里。

请注意这里的重点不是 `ServiceType` 有多复杂，而是：

> 编译期读到的 `google.api.http`，被写进了生成代码，所以运行时还能读到它。

如果没有这一步，运行时的 `HttpTransport` 就不知道应该请求哪个 URL。

---

## 9. 编译期第四步：生成 HTTP Client

HTTP client 生成器是：

- `packages/plugin/src/code-gen/service-client-generator-http.ts`

它主要生成两类代码。

第一类是接口：

```ts
export interface IRumClient {
  report(input: ReportRequest, options?: HttpOptions["requestOptions"]): HttpResult<ReportReply>;
}
```

接口的价值是给业务代码提供清晰的类型约束：

- 调用 `report()` 时必须传 `ReportRequest`。
- 返回结果是 `HttpResult<ReportReply>`。
- 方法级配置只能传 `HttpOptions["requestOptions"]`。

第二类是实现类：

```ts
export class RumClient implements IRumClient, ServiceInfo {
  methods = Rum.methods;
  public defHttp: HttpTransport;

  constructor(vAxios: VAxios | VAxiosInstance, opt: HttpOptions = {}) {
    this.defHttp = new HttpTransport(vAxios, opt);
    this.report = this.report.bind(this);
  }

  report(input: ReportRequest, options?: HttpOptions["requestOptions"]): HttpResult<ReportReply> {
    const method = this.methods[0], opt = options;
    return this.defHttp.request(method, input, opt);
  }
}
```

这段代码的设计很薄。它不自己拼 URL，也不直接调用 axios，而是做三件事：

- 保存当前 service 的 `methods` 元信息。
- 持有一个 `HttpTransport`。
- 每个方法只把 `method + input + options` 转交给 `HttpTransport.request()`。

可以这样讲：

> 生成的 client 是类型安全的门面，真正把元信息翻译成 HTTP 请求的是 `runtime-http`。

---

## 10. 编译期第五步：生成聚合入口 HttpClient

当一个项目里有多个 service 时，如果每个 service 都让业务自己 new，会比较散。

所以 `only_http` 还会额外生成：

- `http-client.ts`

形态类似：

```ts
export class HttpClient {
  rum: RumClient;
  rum1: Rum1Client;

  constructor(vAxios: VAxios | VAxiosInstance, opt: HttpOptions = {}) {
    this.rum = new RumClient(vAxios, opt);
    this.rum1 = new Rum1Client(vAxios, opt);
  }
}
```

它的作用是提供统一入口：

```ts
const httpClient = new HttpClient(vAxios);
httpClient.rum.report(input);
```

这就是 SDK 的常见外观层。业务方只关心 `httpClient.xxx.yyy()`，不需要理解每个 service client 是怎么组装的。

---

## 11. 运行时第一步：业务代码调用生成方法

假设业务代码写：

```ts
httpClient.rum.report({ name: "demo" });
```

实际会进入生成出来的 `RumClient.report()`。

它内部大概做：

```ts
const method = this.methods[0], opt = options;
return this.defHttp.request(method, input, opt);
```

这里的 `method` 不是普通字符串，而是一个方法元信息对象。

它里面包含：

- 方法名
- 输入输出类型名
- 所属 service
- `options["google.api.http"]`

也就是说，到这一步时，业务输入和 proto 注解已经汇合到一起了。

---

## 12. 运行时第二步：HttpTransport 计算 URL

`HttpTransport` 在：

- `packages/runtime-http/src/rpc-transport.ts`

它的 `makeUrl()` 会优先读取：

```ts
method.options["google.api.http"]
```

然后按顺序取其中存在的路由字段：

- `get`
- `post`
- `put`
- `delete`
- `patch`

例如：

```ts
{ post: "/v1/rum/report", body: "*" }
```

最终 URL 就是：

```text
/v1/rum/report
```

如果没有 `google.api.http`，当前实现会退回到：

```text
service.typeName/methodName
```

但在正常 `only_http` 用法里，主路径应该是由 `google.api.http` 明确提供 URL。

---

## 13. 运行时第三步：HttpTransport 计算 HTTP Method

`makeMethod()` 做的事情很直接。

它还是读：

```ts
method.options["google.api.http"]
```

然后做映射：

```text
get    -> GET
post   -> POST
put    -> PUT
delete -> DELETE
patch  -> PATCH
```

所以 proto 里写：

```proto
option (google.api.http) = {
  post: "/v1/rum/report"
  body: "*"
};
```

运行时就会得到：

```text
method = POST
url = /v1/rum/report
```

这就是 `google.api.http` 从“proto 注解”变成“HTTP 请求配置”的关键过程。

---

## 14. 运行时第四步：普通请求和上传请求分流

当前实现里有一个项目约定：

```text
google.api.http.body === "file"
```

代表这是上传接口。

如果是上传接口，生成器会把方法入参类型改成 `UploadFile`，运行时会调用：

```ts
vAxios.uploadFile()
```

并设置：

```text
Content-type: multipart/form-data;charset=UTF-8
```

如果不是上传接口，运行时会调用：

```ts
vAxios.request()
```

并把 input 放到请求配置的 `params` 字段里。

这里要注意：

> `body === "file"` 是这个项目里 HTTP mode 的约定，不是 `google.api.http` 标准语义里的通用上传规范。

这句话很重要，因为它能说明你理解了“标准协议”和“项目定制”的边界。

---

## 15. 运行时第五步：真正发请求的是外部 vAxios

`runtime-http` 自己不是 axios，也不是 fetch。

它要求外部传入一个对象或构造器，满足类似下面的能力：

```ts
interface VAxiosInstance {
  request<T>(config: Record<string, any>, options?: Record<string, any>): HttpResult<T>;
  uploadFile<T>(config: Record<string, any>, params?: UploadFile): HttpResult<T>;
}
```

所以分层是这样的：

```text
生成的 client
  -> 负责类型安全和方法入口

HttpTransport
  -> 负责把 MethodInfo 翻译成 HTTP 请求配置

vAxios
  -> 负责真正发网络请求
```

这也是为什么 `runtime-http` 更像 transport adapter，而不是完整 HTTP 客户端库。

---

## 16. 一次调用的完整链路

以 `Report` 方法为例，完整链路可以这样讲：

```text
1. proto 里定义 rpc Report(ReportRequest) returns(ReportReply)
2. 方法上写 google.api.http，声明 POST /v1/rum/report
3. protoc 把 proto 解析成 CodeGeneratorRequest
4. @protobuf-ts/plugin 读取描述符和 options
5. only_http 分支生成 message.ts、service.ts、service.client.ts、http-client.ts
6. service.ts 里保存 Report 方法的 google.api.http 元信息
7. service.client.ts 里生成 RumClient.report(input)
8. 业务代码调用 httpClient.rum.report(input)
9. report() 取出 Rum.methods[0]，交给 HttpTransport.request()
10. HttpTransport 从 method.options["google.api.http"] 取 URL 和 Method
11. 普通请求调用 vAxios.request()，上传请求调用 vAxios.uploadFile()
12. 返回 HttpResult<ReportReply>
```

如果你能把这 12 步顺下来，就已经能把原理讲得比较完整。

---

## 17. 和标准 protobuf-ts RPC 模式的区别

标准模式关注的是 protobuf/RPC 语义。

它通常关心：

- `MessageType`
- protobuf 二进制编解码
- JSON 编解码
- `runtime-rpc`
- unary / streaming 调用形态
- gRPC / grpc-web / Twirp transport

`only_http` 关注的是前端 HTTP SDK。

它更关心：

- TypeScript 请求类型和响应类型
- `google.api.http` 里的 URL 和 Method
- 普通 HTTP 请求库的接入
- 轻量的 service client
- 业务返回包装 `HttpResult<T>`

可以这样对比：

| 对比项 | 标准模式 | only_http 模式 |
| --- | --- | --- |
| 目标 | protobuf RPC 调用 | 普通 HTTP SDK |
| 路由来源 | RPC service/method 约定 | `google.api.http` |
| 运行时 | `runtime` + `runtime-rpc` | `runtime-http` |
| 网络层 | gRPC / grpc-web / Twirp transport | 外部注入的 `vAxios` |
| 消息能力 | 完整 protobuf 编解码 | 主要保留类型价值 |
| 返回模型 | RPC call / response 抽象 | `HttpResult<T>` |

一句话总结：

> 标准模式是“把 proto 当 RPC 协议跑起来”，`only_http` 是“把 proto 当 HTTP SDK 的契约源来生成代码”。

---

## 18. 为什么不直接复用 runtime-rpc

初学者可能会问：既然仓库已经有 `runtime-rpc`，为什么 `only_http` 还要加 `runtime-http`？

原因是两边抽象目标不同。

`runtime-rpc` 面向的是 RPC 调用模型。它要支持：

- unary
- server streaming
- client streaming
- duplex streaming
- RPC metadata
- RPC status
- 不同 RPC transport 的统一契约

`runtime-http` 面向的是普通 HTTP SDK 调用。它要支持：

- 从 `google.api.http` 取 URL
- 从 `google.api.http` 取 HTTP Method
- 判断是否走上传
- 转交给外部 HTTP 库
- 返回业务包装结构

如果硬把这些 HTTP SDK 逻辑塞进 `runtime-rpc`，会让标准 RPC 抽象变得混乱。

所以当前方案是：

> 标准 RPC 继续走 `runtime-rpc`，HTTP SDK 分支走更薄的 `runtime-http`。

这是一种按目标拆运行时的设计。

---

## 19. 当前实现的边界

讲原理时也要讲边界，否则容易把这个模式说得过度通用。

当前 `only_http` 实现有几个明显边界。

### 19.1 它不是标准 protobuf over HTTP

它没有把请求按完整 protobuf over HTTP 协议来编码，也没有提供完整 RPC 生命周期。

它更像：

```text
proto 契约 + HTTP 注解 -> TypeScript HTTP SDK
```

### 19.2 它没有完整实现 google.api.http 的所有语义

当前主要用到：

- `get`
- `post`
- `put`
- `delete`
- `patch`
- `body`

更复杂的能力，比如路径变量替换、`additional_bindings`、细粒度 body 字段映射，在当前实现里不是完整主线。

### 19.3 普通请求统一放到 params

`HttpTransport.request()` 里普通请求会把输入放到：

```ts
params: input
```

这说明它没有严格区分 `body: "*"` 时到底应该放 body 还是 query，而是把解释权更多交给下层 `vAxios` 或团队约定。

### 19.4 上传规则是项目约定

`body === "file"` 被用来判断上传。

这很实用，但它不是标准 `google.api.http` 里天然规定的通用语义。

## 20. 新人最容易混淆的几个点

### 20.1 only_http 不是只生成 HTTP 请求函数

它还要生成：

- 消息接口
- 枚举
- 服务元信息
- service client
- 聚合 client

因为一个完整 SDK 不只是请求函数，还需要类型和元信息。

### 20.2 google.api.http 不是运行时才去读 proto

运行时不会读 `.proto` 文件。

编译期已经把 `google.api.http` 写进生成的 `service.ts`，运行时只是读取这个 TypeScript 对象。

### 20.3 ServiceType 不是用来发请求的

`ServiceType` 保存服务和方法元信息。

真正发请求的入口是 `HttpTransport.request()`，真正网络请求由 `vAxios` 完成。

### 20.4 生成的 client 不等于网络层

生成的 client 只是强类型门面。它知道有哪些方法、方法入参出参是什么、哪个 `MethodInfo` 对应哪个方法。

网络细节被下沉到 `runtime-http` 和外部 `vAxios`。

### 20.5 only_http 不是标准 gRPC 的变体

它不是“把 gRPC transport 换成 HTTP”。

更准确地说，它是“复用 proto 契约和 protobuf-ts 生成框架，生成普通 HTTP SDK”。

## 21. 面试式问答

### Q1：only_http 的输入和输出是什么？

输入是 `.proto` 契约，尤其是 service、message、enum 和方法上的 `google.api.http` 注解。

输出是 TypeScript 代码，包括消息接口、服务元信息、HTTP client 和聚合 `HttpClient`。

### Q2：only_http 为什么依赖 google.api.http？

因为普通 HTTP 调用需要 URL 和 HTTP Method，而纯 RPC 方法只描述输入输出，不描述 REST 路由。`google.api.http` 正好把 RPC 方法映射到 HTTP 路由。

### Q3：运行时为什么还能拿到 google.api.http？

因为编译期生成 `service.ts` 时，把 `google.api.http` 写进了 `ServiceType.methods[].options`。运行时读取的是生成后的 TypeScript 元信息对象，不是读取 proto 文件。

### Q4：HttpTransport 的职责是什么？

它接收 `MethodInfo + input + options`，从 `MethodInfo.options["google.api.http"]` 里计算 URL 和 Method，判断是否上传，然后调用外部注入的 `vAxios`。

### Q5：vAxios 是什么？

它是项目要求外部提供的 HTTP 实现，可以是某个 axios 封装，也可以是符合接口的其他对象。`runtime-http` 不自己绑定具体网络库。

### Q6：only_http 和标准 protobuf-ts RPC 最大区别是什么？

标准模式是完整 protobuf/RPC 运行链路，强调消息编解码和 RPC transport；`only_http` 是 HTTP SDK 生成链路，强调从 proto 注解生成普通 HTTP 调用入口。

### Q7：为什么说 only_http 是定制分支？

因为它有明显业务约定，例如 `HttpResult<T>` 的返回包装、`body === "file"` 的上传判断、普通请求统一走 `params`。这些都不是通用 protobuf RPC 标准的一部分。

## 22. 可以直接复述的 30 秒版本

`only_http` 是 `protobuf-ts` 插件里的一个生成分支。它复用 `protoc` 和描述符解析能力，读取 proto 里的 service、message、enum，以及方法上的 `google.api.http` 注解，然后生成 TypeScript 消息类型、HTTP 服务元信息、服务 client 和聚合 `HttpClient`。运行时业务调用生成的 client 方法，client 把对应 `MethodInfo` 交给 `HttpTransport`，`HttpTransport` 再根据 `google.api.http` 算出 URL 和 HTTP Method，最后委托外部注入的 `vAxios` 发请求。

## 23. 可以直接复述的 3 分钟版本

`protobuf-ts` 本质上分两段：编译期和运行时。编译期由 `protoc` 把 `.proto` 解析成 `CodeGeneratorRequest`，`@protobuf-ts/plugin` 再遍历描述符生成 TypeScript 文件。运行时则由生成代码配合 runtime 包完成消息或 RPC 调用。

普通模式下，`protobuf-ts` 更接近标准 protobuf/RPC 工具链，会生成 `MessageType`、标准 `ServiceType`、generic client、grpc client 等，然后通过 `runtime`、`runtime-rpc` 和具体 transport 去跑 gRPC、grpc-web 或 Twirp。

`only_http` 不走这个目标。它通过 `--ts_opt=only_http` 打开，生成器会切到 HTTP 专用分支。这个分支仍然生成消息接口和枚举，但服务部分会生成 HTTP 专用 `ServiceType` 和 HTTP client。最关键的是，方法上的 `google.api.http` 注解会被写进生成的服务元信息里，例如 `options["google.api.http"] = { post: "/v1/rum/report", body: "*" }`。

运行时业务调用 `httpClient.rum.report(input)` 时，生成的 `RumClient.report()` 不自己拼 URL，而是取出对应的 `MethodInfo`，调用 `HttpTransport.request(method, input, options)`。`HttpTransport` 从 `method.options["google.api.http"]` 里判断 URL、HTTP Method 和是否上传，然后调用外部传入的 `vAxios.request()` 或 `vAxios.uploadFile()`。

所以 `only_http` 的本质不是标准 gRPC transport，也不是简单 axios 封装，而是“用 proto 作为接口契约源，用 `google.api.http` 作为 HTTP 路由源，自动生成类型安全的前端 HTTP SDK”。

---

## 24. 最后记住这三句话

1. `only_http` 的核心价值是把 proto 契约稳定翻译成 TypeScript HTTP SDK。
2. `google.api.http` 是连接 RPC 方法和普通 HTTP 路由的桥。
3. 生成的 client 只是类型安全门面，真正把元信息变成 HTTP 请求的是 `runtime-http` 的 `HttpTransport`，真正发网络请求的是外部注入的 `vAxios`。
