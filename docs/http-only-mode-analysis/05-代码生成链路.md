# 代码生成链路

## 1. 生成哪些文件

在 `only_http` 模式下，核心生成物有四类：

- 消息接口文件
- 枚举文件
- 服务元信息文件
- HTTP 客户端文件

以 `packages/example-http` 为例：

- `message.ts`
- `service.ts`
- `service.client.ts`
- `service1.ts`
- `service1.client.ts`
- `http-client.ts`

## 2. 消息接口仍然保留

即使启用了 `only_http`，消息接口和枚举仍然会生成。

这是因为前端 SDK 仍然需要：

- 请求参数类型
- 响应数据类型
- 枚举定义

但这个模式不会重点生成标准 `MessageType` 运行时代码。

换句话说，这里保留的是“静态类型价值”，不是“完整 protobuf 编解码能力”。

## 3. ServiceTypeGeneratorHttp

HTTP 模式下的服务元信息生成器是：

- `packages/plugin/src/code-gen/http-type-generator.ts`

它会生成类似这样的代码：

```ts
export const Rum = new ServiceType("rum.Rum", [
  {
    name: "Report",
    options: { "google.api.http": { post: "/v1/rum/report", body: "*" } },
    I: { typeName: "ReportRequest" },
    O: { typeName: "ReportReply" }
  }
]);
```

这里的重点不是 `ServiceType` 本身有多复杂，而是它把方法的 HTTP 元信息保存在运行时可读对象里。

## 4. ServiceClientGeneratorHttp

HTTP 客户端生成器是：

- `packages/plugin/src/code-gen/service-client-generator-http.ts`

它主要做三件事：

1. 注册符号
2. 生成接口
3. 生成实现类

### 4.1 注册符号

每个 service 会生成两种主要命名：

- `IxxClient`
- `xxClient`

例如：

- `IRumClient`
- `RumClient`

这样做是为了让后续 import、引用和聚合类生成时都能稳定定位。

### 4.2 生成接口

接口是对外暴露的类型层 API。

普通方法会生成：

```ts
report(input: ReportRequest, options?: HttpOptions["requestOptions"]): HttpResult<ReportReply>;
```

上传方法会生成：

```ts
transforming(input: UploadFile, options?: HttpOptions["requestOptions"]): HttpResult<ReportReply>;
```

其中，是否是上传方法由下面规则决定：

- `method.options["google.api.http"].body === "file"`

### 4.3 生成实现类

实现类内部结构非常固定：

```ts
export class RumClient implements IRumClient, ServiceInfo {
  methods = Rum.methods;
  public defHttp: HttpTransport;

  constructor(vAxios, opt = {}) {
    this.defHttp = new HttpTransport(vAxios, opt);
    this.report = this.report.bind(this);
  }

  report(input, options) {
    const method = this.methods[0], opt = options;
    return this.defHttp.request(method, input, opt);
  }
}
```

这说明生成类并不自己处理复杂协议，只是：

- 保存方法元信息
- 保存 transport
- 把调用转发给 `HttpTransport.request()`

## 5. 聚合类 HttpClient

生成器还会额外生成一个 `HttpClient` 聚合类。

它的作用是：

- 统一挂载多个 service client
- 为业务提供一个总入口

例如：

```ts
export class HttpClient {
  rum: RumClient;
  rum1: Rum1Client;

  constructor(vAxios, opt = {}) {
    this.rum = new RumClient(vAxios, opt);
    this.rum1 = new Rum1Client(vAxios, opt);
  }
}
```

这相当于给生成 SDK 提供了一个门面层。

## 6. 生成阶段的关键结论

`only_http` 模式的生成阶段，本质上是：

1. 先把 proto service 解释成带选项的服务元信息
2. 再把这些元信息转成薄包装 HTTP Client
3. 最终把实际请求逻辑下沉到 `runtime-http`

所以它的核心价值不在“生成多少代码”，而在“把 proto 注解稳定翻译成类型安全的调用入口”。
