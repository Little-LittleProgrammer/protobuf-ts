# only_http 模式总览

## 1. 项目定位

`protobuf-ts` 不是单一库，而是一组围绕 `Protocol Buffers` 和 `RPC` 的 TypeScript 工具链集合。

从仓库结构上看，它主要包含三类能力：

- 编译期能力：把 `.proto` 交给 `protoc` 后，生成 TypeScript 代码。
- 运行时能力：提供消息类型、反射信息、RPC 抽象和若干 transport。
- 示例工程：验证生成物的形态和实际接入方式。

仓库总说明可先看根目录的 `README.md`。

## 2. only_http 模式是什么

`only_http` 是 `@protobuf-ts/plugin` 里的一个生成开关，不是独立插件。

它的目标不是生成标准 gRPC / Twirp 客户端，而是把：

- `proto service`
- `google.api.http` 方法注解

转成一套适合前端业务调用的 HTTP SDK。

也就是说，`only_http` 更像：

`proto 契约 -> 前端 HTTP Client 生成器`

而不是：

`proto 契约 -> 标准 protobuf RPC transport`

## 3. only_http 的核心思路

在这个模式下，系统做了三件事：

1. 读取 `.proto` 里的 `service`、`message`、`enum` 和 `google.api.http` 选项。
2. 为每个 `service` 生成一个轻量的 TypeScript HTTP Client。
3. 在运行时根据方法元信息，把调用转成普通 HTTP 请求。

## 4. 与标准 protobuf-ts RPC 链路的区别

标准 RPC 链路一般是：

- 生成 `MessageType`
- 生成标准 `ServiceType`
- 使用 `runtime-rpc`
- 搭配 gRPC / Twirp / grpc-web transport

而 `only_http` 的链路是：

- 仍然生成消息接口和枚举
- 不再重点依赖标准 protobuf 编解码运行时
- 生成 HTTP 专用 `ServiceType`
- 生成 HTTP Client
- 调用 `runtime-http` 的 `HttpTransport`

## 5. only_http 的最终产物

以 `packages/example-http` 为例，生成结果主要有：

- `message.ts`
  - 消息接口和枚举
- `service.ts`
  - 服务元信息
- `service.client.ts`
  - 单个服务对应的 HTTP Client
- `http-client.ts`
  - 多服务聚合入口

## 6. 一句话总结

`only_http` 的本质，是在 `protobuf-ts` 标准代码生成链路之外，额外分出一条“基于 proto 注解生成 REST 风格前端 SDK”的实现路径。
