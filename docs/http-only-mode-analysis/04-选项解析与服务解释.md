# 选项解析与服务解释

## 1. 为什么 google.api.http 是核心

`only_http` 模式真正依赖的不是 service 名字本身，而是方法上的 `google.api.http` 注解。

例如 `packages/example-http/protos/service.proto` 里：

```proto
rpc Report(ReportRequest) returns(ReportReply) {
  option (google.api.http) = {
    post: "/v1/rum/report",
    body: "*",
  };
}
```

如果没有这类注解，运行时就拿不到目标 URL 和 HTTP Method。

## 2. 选项为什么要“重新解释”

在 protobuf 描述符里，自定义扩展选项通常并不是一开始就以结构化对象存在。

很多情况下，它们先挂在 `descriptor.options` 的 unknown fields 中。

所以插件想真正拿到：

```json
{
  "google.api.http": {
    "post": "/v1/rum/report",
    "body": "*"
  }
}
```

就必须做一次“扩展还原”。

## 3. Interpreter.readOptions() 的思路

`packages/plugin/src/interpreter.ts` 的 `readOptions()` 做了这件事。

它的核心流程是：

1. 检查当前描述符有没有 `options`
2. 从 `options` 中取 unknown fields
3. 判断当前描述符对应的是哪类 options
   - `MethodOptions`
   - `FieldOptions`
   - `ServiceOptions`
   - `MessageOptions`
   - `FileOptions`
4. 在注册表中查找该 options 类型上的所有扩展字段
5. 构造一个 synthetic message type
6. 用这个 synthetic type 重新读取 unknown bytes
7. 再把结果转成 JSON

## 4. 这一步为什么重要

只有做完这一步，`google.api.http` 才会从“编译器扩展字段”变成“普通 JSON 元信息”。

而后面的生成器实际上只关心 JSON 结果，不关心底层 protobuf 扩展解码细节。

所以可以说：

- `DescriptorRegistry` 负责找出“有哪些扩展”
- `Interpreter.readOptions()` 负责把扩展值真正读出来

## 5. 服务解释

当生成器调用：

```ts
interpreter.getServiceType(descriptor)
```

时，`Interpreter` 会构造出一个标准化的服务对象。

其中每个方法都会被解释成包含以下内容的结构：

- `name`
- `localName`
- `I`
- `O`
- `options`
- 可选的 `idempotency`
- 可选的 streaming 标记

对 `only_http` 来说，重点字段是：

- `I`
  - 输入消息类型
- `O`
  - 输出消息类型
- `options`
  - 其中包含 `google.api.http`

## 6. MethodInfo 对 only_http 的意义

HTTP 生成器后续不会再回头分析原始 `MethodDescriptorProto`。

它只会围绕解释后的 `MethodInfo` 工作：

- 根据 `options["google.api.http"]` 判断 URL 和动词
- 根据 `body === "file"` 判断是否上传
- 根据 `I` / `O` 决定方法签名的输入输出类型

所以 `Interpreter` 在这里承担的是“把描述符翻译成生成器语言”的角色。

## 7. 小结

`only_http` 模式之所以成立，不是因为直接读取 proto 文本，而是因为：

1. protobuf 描述符完整保留了方法选项
2. `DescriptorRegistry` 能找到扩展定义
3. `Interpreter.readOptions()` 能把扩展恢复成 JSON
4. HTTP 生成器再消费这些 JSON 元信息

这四步缺一不可。
