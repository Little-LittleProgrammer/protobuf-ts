# 编译期链路

## 1. 入口命令

`only_http` 模式由 `protoc` 命令触发。

示例见 `packages/example-http/Makefile`：

```makefile
npx protoc --proto_path=./protos \
  --proto_path=./third_party \
  --ts_out=. \
  --ts_opt=only_http \
  --ts_opt=long_type_number \
  --ts_opt=use_proto_field_name \
  $(API_PROTO_FILES)
```

这里真正决定行为的是：

- `--ts_out=.`
- `--ts_opt=only_http`

## 2. PluginBase 如何接住 protoc 请求

`protoc` 不会直接调用业务生成器，而是把请求序列化成 `CodeGeneratorRequest`，通过标准输入传给插件。

`packages/plugin-framework/src/plugin-base.ts` 中的 `PluginBase.run()` 负责：

1. 从 `stdin` 读取二进制请求
2. 反序列化成 `CodeGeneratorRequest`
3. 调用具体插件的 `generate()`
4. 把结果封装成 `CodeGeneratorResponse` 返回给 `protoc`

所以编译期真正的输入并不是 `.proto` 文本本身，而是 protobuf 描述符请求。

## 3. 参数解析

`packages/plugin/src/protobufts-plugin.ts` 的 `generate()` 是主入口。

它首先调用：

```ts
makeInternalOptions(
  this.parseOptions(this.parameters, request.parameter),
  ...
)
```

其中：

- `parseOptions()` 负责把 `request.parameter` 解析成布尔开关集合
- `makeInternalOptions()` 再把它们转换成内部选项对象

`only_http` 参数在 `packages/plugin/src/our-options.ts` 中被定义为：

- 对应外部参数名：`only_http`
- 对应内部开关：`options.onlyHttp`

也就是说，这个模式完全由一个布尔开关控制。

## 4. DescriptorRegistry 的作用

`generate()` 接下来会执行：

```ts
const registry = DescriptorRegistry.createFrom(request);
```

这一步非常关键。

`DescriptorRegistry` 会把：

- `request.protoFile`
- 各文件里的 message / enum / service
- 类型引用关系
- 源码位置信息

整理成统一的描述符视图。

之后所有生成器都不再直接操作原始 `protoFile[]`，而是围绕这个注册表工作。

## 5. Interpreter 的作用

生成器不会直接使用裸 `DescriptorProto` / `ServiceDescriptorProto`。

它们会先通过：

```ts
const interpreter = new Interpreter(registry, options)
```

做一层解释。

`Interpreter` 的职责包括：

- 把消息描述符转换成运行时消息类型信息
- 把服务描述符转换成服务元信息
- 把方法描述符转换成 `MethodInfo`
- 读取自定义选项

对于 `only_http` 模式来说，最重要的是最后一项：读取 `google.api.http`。

## 6. 生成分支切换

当遍历到 service 时，`ProtobuftsPlugin.generate()` 会根据 `options.onlyHttp` 分支：

- 普通模式：走标准 `ServiceTypeGenerator`、`ServiceClientGeneratorGeneric` 等
- `only_http` 模式：走 `ServiceTypeGeneratorHttp` 和 `ServiceClientGeneratorHttp`

这一步说明：

`only_http` 不是重新实现整个插件，而是在最终输出阶段切换生成器。

## 7. 编译期链路总结

编译期完整链路可以概括为：

1. `protoc` 解析 `.proto`
2. 生成 `CodeGeneratorRequest`
3. `PluginBase.run()` 读取请求
4. `ProtobuftsPlugin.generate()` 解析参数
5. `DescriptorRegistry` 组织所有描述符
6. `Interpreter` 解释服务与选项
7. 根据 `only_http` 开关，选择 HTTP 专用生成器
8. 输出 `.ts` 文件
