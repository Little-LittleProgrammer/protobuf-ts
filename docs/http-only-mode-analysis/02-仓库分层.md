# 仓库分层

## 1. 总体结构

这个仓库是一个 monorepo，核心目录集中在 `packages/`。

理解 `only_http` 模式时，最重要的是下面几层：

- `packages/plugin-framework`
- `packages/plugin`
- `packages/runtime`
- `packages/runtime-rpc`
- `packages/runtime-http`
- `packages/example-http`

## 2. plugin-framework

`plugin-framework` 是 `protoc` 插件的基础设施层。

主要职责：

- 读取 `CodeGeneratorRequest`
- 管理描述符树
- 管理类型名与源码位置信息
- 提供 TypeScript AST 输出能力

关键文件：

- `packages/plugin-framework/src/plugin-base.ts`
- `packages/plugin-framework/src/descriptor-registry.ts`

其中：

- `PluginBase` 负责插件运行入口、参数解析、响应输出。
- `DescriptorRegistry` 负责把所有 `FileDescriptorProto` 组织成一个可遍历、可解析的描述符注册表。

## 3. plugin

`plugin` 是真正的 `protoc-gen-ts` 实现。

主要职责：

- 解析命令行参数
- 创建 `Interpreter`
- 遍历描述符
- 按不同模式生成 TS 文件

`only_http` 的核心逻辑基本都在这一层。

关键文件：

- `packages/plugin/src/protobufts-plugin.ts`
- `packages/plugin/src/our-options.ts`
- `packages/plugin/src/interpreter.ts`
- `packages/plugin/src/code-gen/http-type-generator.ts`
- `packages/plugin/src/code-gen/service-client-generator-http.ts`

## 4. runtime

`runtime` 是标准 protobuf 消息运行时。

主要职责：

- 二进制编解码
- JSON 编解码
- 反射创建、比较、类型检查

这一层在标准 `protobuf-ts` 工作流里非常重要，但在 `only_http` 模式下会被明显弱化，因为 `only_http` 主要生成静态消息接口，而不是完整 `MessageType`。

## 5. runtime-rpc

`runtime-rpc` 是标准 RPC 抽象层。

主要职责：

- 定义 `ServiceInfo` / `MethodInfo`
- 定义 unary / streaming 调用抽象
- 为 transport 提供统一接口

gRPC、grpc-web、Twirp 这些 transport 都主要围绕它展开。

## 6. runtime-http

`runtime-http` 是 `only_http` 专用运行时。

它的抽象比 `runtime-rpc` 更薄，重点是：

- `ServiceType`
- `MethodInfo`
- `HttpTransport`
- `HttpResult`
- `HttpOptions`

它并不尝试实现完整 protobuf RPC 语义，而是把方法调用转成普通 HTTP 请求。

## 7. example-http

`packages/example-http` 是理解 `only_http` 的最佳样例。

它展示了：

- 如何写带 `google.api.http` 的 proto
- 如何通过 `--ts_opt=only_http` 生成代码
- 最终输出长什么样

这个示例比单看设计文档更有价值，因为它直接对应真实生成结果。

## 8. 分层关系

如果把 `only_http` 模式按层串起来，可以理解为：

1. `plugin-framework` 提供插件基础设施。
2. `plugin` 解析参数、解释描述符、生成代码。
3. `runtime-http` 承接运行时调用。
4. `example-http` 验证端到端输出。

这条链路里，`plugin` 是编译期中心，`runtime-http` 是运行时中心。
